§ 00 · ÖVERSIKTVad det här dokumentet är

Designen i katalogen visar hur Sadaqa Sweden ska kännas. Det här dokumentet säger vad som saknas och exakt hur det ska byggas för att kännas så på riktigt — i kod, i alla tillstånd, i alla flöden.

Dokumentet är skrivet för Claude Code att läsa rakt igenom. Det är två delar:

  1. Designgenomgång — det jag (designern) ser brister i nu när jag tittar på mitt eget arbete. Per yta, samt genomgripande mönster.
  2. Byggplan — arkitektur, filstruktur, exakta routes, komponentträd, tillståndshantering, tredjeparts-integrationer, byggsekvens.
Att hålla i minnet

Designen är inte färdig — den är en grund. Det här dokumentet ska inte tolkas som att Claude Code måste replikera HTML-katalogen pixel för pixel. Designen pekar ut känsla, tonen och layoutprincipen. Byggplanen säger hur den känslan ska realiseras med riktiga komponenter, riktiga states, och riktig data.

Allt refererar tillbaka till de 17 modulerna (M1–M17) i 1-Planering/. När en modulnotation som M4 dyker upp betyder det att modul-filen är källan för affärsregler — designen och byggplanen är tolkningen.

§ 01 · DESIGN-GENOMGÅNGPer yta — vad jag faktiskt missade

Tre kategorier av fynd: mindre (kosmetiskt), fixa (gör innan launch), kritiskt (utan detta är ytan trasig).

Mindre — efter launch
Fixa — innan launch
Kritiskt — utan detta är ytan trasig
A1

Marknad — startsidan

marketing.html
"8,4 mkr samlat in" är aspirationellt — inte sant vid launch. Plattformen startar med 0. Hero-stat-stripen kommer kännas falsk när första besökaren landar.
Bygg en StatStrip-komponent som läser riktiga värden från donation.sum(). Vid 0 visa istället ett "Vi startar nu"-meddelande eller dölj stripen. Lägg en cron som uppdaterar siffran realtid.
"100 % till mottagaren" är inte sant. Stripe tar 1,4 % + 1,80 kr. Det är inte 100 %, det är 98,5 %.
Ändra copy: "0 % plattformsavgift — vi finansieras inte av era donationer. Betalanbjudaren (Stripe) tar 1,4 % + 1,80 kr per donation, det enda som dras innan beloppet når mottagaren." Tydligare och håller juridiskt.
Inga riktiga foton. Hela hero, alla campaign-cards, allt är gradienter. Vid launch kommer detta att kännas billigt.
Bestäm en bildstrategi: (a) Inköpta stockfoton för marknadsföring, (b) endast bilder från publicerade insamlingar visas — om 0 insamlingar publicerade, visa en minimalistisk illustration istället för fake-bilder.
Quote-sektionens hadith är inte verifierad. "Den som visar vägen till något gott..." — referensen är Muslim 1893 men jag har inte dubbelkollat. Att citera en hadith felaktigt på en muslimsk plattform vore förtroendebrist.
Låt Ahmed N. (fiqh-granskaren) verifiera alla religiösa citat innan publicering. Lista godkända citat i lib/content/citat.ts.
Hero-kortet roterar -2°. Snyggt men kan kännas pretentiöst för en seriös sajt.
Testa A/B — rotation av eller 0,5°. Behåll om det får sidan att kännas "premium-redaktionell" snarare än "startup-cool".
A2

Discovery — listning

discovery.html
Range-slidern för belopp är bara visuell. Filtret är inte funktionellt i designen, men användare förväntar sig att det fungerar.
Använd <input type="range"> dubbelt med JS, eller bygg en debounced React-state. Skicka URL-params ?amt_min=10000&amt_max=200000 och filtrera serversidigt på en query.
14 resultat visas men 9 syns. Ingen paginering — bara en "Visa nästa 9" som inte gör något.
Använd cursor-baserad paginering (created_at + id) eller infinite scroll med useIntersectionObserver. Server action returnerar nästa 12.
Inget empty state. Vad ser användaren om filtren inte ger något resultat?
Designa en NoResults-komponent: stor "Inget matchar." + förslag att (a) ta bort filter, (b) bläddra hela kartan, (c) sätta upp en bevakning ("notifiera mig när en insamling matchar").
"Zakat-OK"-filter behöver förklaring. Många (inkl muslimer) vet inte exakt vad som kvalificerar för zakat. Pillen är där men en tooltip saknas.
Lägg en Popover bredvid filter-rubriken som öppnar en kort förklaring (2 meningar) + länk till "Sadaqa & Zakat"-sidan.
A3

Insamlingssidan

fundraiser.html
Tabs gömmer kritiskt innehåll. "Mottagare & pengaflöde", "Granskning", "Konversation" är alla bakom tabs som inte fungerar — och dessa är de viktigaste sakerna för donatorns förtroende.
Två alternativ: (a) Slopa tabs, visa allt i en lång scroll. (b) Behåll tabs men förladda alla sektioner i DOM så Ctrl+F hittar dem. Jag rekommenderar (a) — förtroende byggs genom att se allt utan att klicka.
Side-panelen (donera) skalar inte med scroll. När man scrollar ner till botten av storyn försvinner CTA-knappen. På långa sidor donerar folk inte.
Sticky-position på donate-card men med max-height = viewport - 96px. Vid scroll på mobil: byt mot en fixed bottom bar med "Donera"-knapp.
Inga foton. Hero-bilden och thumbnails är gradienter. Här är det extra känsligt — donatorer som inte ser verkliga människor de hjälper donerar mindre.
Min 1 bild av mottagar-platsen eller liknande projekt. Insamlaren laddar upp bilder i wizarden (steg 2). Granskaren verifierar EXIF + omvänd bildsök innan publicering.
Tre obligatoriska bevis visas på en gång — det förvirrar pre-utbetalning. När en insamling just publicerats finns bara bevis 1 (eller 0). Tidlinjen visar alla tre som "väntar" vilket ser ut som ofullständigt arbete.
Visa bara bevis som faktiskt finns, plus en diskret rad längst ner som säger "2 bevis till kommer — vid utbetalning och vid resultat". Inte tre tomma kort.
"Senaste donatorer" visar 4. Skulle kunna vara live (websocket-uppdaterad).
Använd Supabase Realtime + donation-tabellen. Animera in nya rader med fade-in. Begränsa till 10 senaste.
A4

Donator-flödet

donate.html
Inget felfall. Vad händer om Stripe avvisar kortet? Om nätverket dör? Om beloppet ändrades sedan man valde det? Designen har bara "happy path".
Bygg PaymentError-komponent. Tre subtyper: card_declined (visa Stripe-felmeddelande + behåll övriga fält), network_error (retry-knapp), amount_changed (visa nytt belopp + bekräfta). Skriv felmeddelanden på svenska.
Anonymitet är otydligt. "Donera anonymt" — anonym för vem? Insamlaren? Andra donatorer? Skatteverket?
Bryt upp i tre nivåer: (1) "Publikt namn dolt — bara förnamn", (2) "Anonym för insamlaren — bara plattformen ser dig", (3) "Helt anonym — ingen ser namn (men kvitto i din mejl kvarstår, lagkrav)". Skatteverket-rad: "Avdrag möjligt? Då måste namnet kunna kopplas till ditt personnummer."
"Månatlig donation" — Stripe-subscriptions är annorlunda än one-time. Behöver ett Stripe Customer (sparas), kortet sparas (PCI-implications), webhook för failed renewals.
Lägg månatlig bakom en feature-flag och bygg inte i v1. Eller bygg på riktigt med Stripe Subscriptions + kund-portal för att avbryta. Mocka inte.
Swish-flödet är inte ritat. "Swish · QR-kod öppnas i nästa steg" säger designen — men inget steg 5.
Swish behöver en QR-poll-state: visa QR + spinner + status "Väntar på betalning..." → "Klart!" eller timeout. Använd Stripes Swish-integration (PaymentMethod via PaymentIntent).
Confetti-animation på success? Inte i designen — men många plattformar lägger till. För muslimsk kontext är det troligen fel ton.
Skip confetti. Den nuvarande success-screen med stilla "Jazak Allah khair" är rätt ton — håll den värdig.
A5

Skapa-wizarden

wizard.html
Stripe Connect-onboarding saknas helt. Designen antar att insamlaren har Stripe-konto. I verkligheten måste vi öppna det åt dem (eller länka till Stripes Express-onboarding) innan första donation kan tas emot.
Lägg ett extra steg 5,5 efter granskning godkänts: "Innan din insamling kan ta emot betalningar behöver du koppla ett Stripe-konto." Använd Stripe Connect Express-onboarding flow — Stripe hostar formuläret, vi får ett connected_account_id tillbaka via webhook.
Live-preview uppdateras inte i designen. Den är statisk. Användaren skriver men ser inte sin titel/bild förändras.
Bygg WizardPreview som lyssnar på samma React-state som formuläret. Använd useDeferredValue för att inte rerender per keystroke.
Bilduppladdning är inte ritad — inga progress bars, ingen klippning.
Använd react-image-crop eller liknande. Tvinga 16:9 för omslagsbild, 1:1 för thumbnails. Spara till R2 via signed URL från en server action. Visa progress per bild.
Validation per steg saknas. Användaren kan gå till granska-steget utan att fylla i story-fältet. Designen säger inte hur fel visas.
Använd zod för schema-validering per steg. Visa inline-fel med röd ram + meddelande under fältet. Disable "Nästa"-knappen tills steget validerar.
"Per-fält integritet" är ett unikt koncept som behöver mer förklaring. Använda ser tre toggles och vet inte vad som händer.
Lägg en kort intro-paragraf ovanför: "Du styr exakt vad som syns publikt. Granskaren ser allt, donatorerna ser bara det du gjort publikt. Detta är vår 'per-fält-integritet'-princip."
A6

Granskningsvy

review.html
Eskalering-flödet är otydligt. "Väntar på Salma K." — vad gör Zivar om Salma är offline 3 dagar? SLA fortsätter ticka. Designen säger inte hur Zivar kan eskalera till nästa granskare.
Lägg en "Eskalera"-knapp på principen-raden: Stora belopp >100k. Den öppnar en modal: "Hoppa över Salma (offline 18h) — ta in Ahmed Z. istället". Logga eskaleringen i granskning_handelse.
"Avslå"-knappen behöver tvinga motivering + bekräftelse. Just nu är det en stor röd knapp som ser ut som att den kan klickas av misstag.
Bygg RejectModal: kräver fritext min 80 tecken + dropdown med skäl (bedrägeri, icke-islamiskt, dubblett, otydligt mål, mottagare overifierbar). Inga snabb-skäl — varje avslag motiveras öppet.
Dokument-pdf:er visas inte inline. Granskaren måste ladda ner varje PDF för att se den.
Bygg in react-pdf så PDF:er renderas inline med zoom + sidnavigering. Annoteringar valfritt i v1 — bara visning räcker.
Auto-verifiering av dokument är magiskt — vad verifieras egentligen?
Definiera tre auto-checks: (1) Stripe-transferkvitto verifieras mot Stripe API (är denna transfer-ID äkta?), (2) EXIF + omvänd bildsök via TinEye eller Bing-API, (3) Organisations-registreringsbevis via Skatteverkets API (för svenska föreningar). Bygg en VerificationLog som visar exakt vad som kollades.
A7

Insamlarens konto

account.html
"Svara granskaren"-knappen leder till granskningsvyn (review.html) — fel publik. Insamlaren ska inte se granskningskön — bara sin egen iteration.
Bygg en separat /insamling/[id]/iteration route. Visar bara konversationen + möjlighet att svara + ladda upp nya dokument. Kalla den för "Granskar-konversation" på insamlarsidan.
Trust-poäng 92 visas för ny användare — det är fel. Ny användare har 0 eller "ej rankad".
Default trust = "Ej rankad ännu (0 insamlingar)" med en länk till "Hur byggs trust-poäng?". Score börjar visas efter första avslutade insamling.
Inga notiser i sidopanelen. "12 meddelanden" är en räknare men ingen ikon-badge eller dropdown.
Bygg NotificationCenter som dropdown från klock-ikonen. Realtid via Supabase Realtime. Tre typer: granskar-respons, ny donator, insamling avslutas snart.
A8

Övriga ytor — batchade fynd

admin · team · map · community · catalog · auth · profile · update
Admin · charten är illustrativ SVG — inte data.
Använd recharts eller visx. Servera data från en server component som queryar donation-tabellen med date_trunc('day', skapad).
Karta · Sverige-SVG är stiliserad — inte geografiskt korrekt.
Hämta GeoJSON för Sveriges kommuner (Lantmäteriet). Använd react-simple-maps eller MapLibre. Aggregera insamlingar per kommun.
Karta · pinning av insamlare på stadsnivå är ett integritetsproblem. Vissa donatorer/insamlare vill inte att deras bostadsort syns.
Default: aggregera på län-nivå. Insamlaren får opt-in för stad-nivå pin. Använd profil.geografisk_synlighet-fält (enum: land · lan · stad).
Community · dua-knappen lagrar vad? Är "142 dua" antal klick eller unika personer? Kan en person klicka två gånger?
En dua per person per post. Unique constraint på (post_id, profil_id). Gäster räknas inte (måste logga in). Visa antal: SELECT COUNT(DISTINCT profil_id).
Auth · BankID-flödet visas som mockup. Det riktiga flödet kräver broker (Criipto eller Freja eID) — det är inte trivialt.
Använd Criipto Verify (smidigast). De ger en iframe + JWT-token tillbaka. Vi mappar JWT → vår profil. PRIVAT-fält: vi sparar aldrig personnumret rådatamässigt — bara en hash + Criiptos sub-claim.
Update · "auto-verifierad" Stripe-kvitto kräver Stripe API-lookup.
När insamlaren laddar upp en pdf, parsa Stripe-transfer-ID:t (regex på text) → query Stripes API → matcha mot förväntat belopp + connected_account. Markera som auto-verifierad i transparens_bevis.verifierings_status.

§ 02 · GENOMGRIPANDEDesignbrister som drabbar alla ytor

Det här är mönstren jag missade som inte tillhör en enskild yta — men som måste lösas i hela plattformen.

G1

Tomma tillstånd (empty states)

Plattformen launchar med 0 data. Inga insamlingar publicerade, inga donatorer, inga föreningar. Alla mina mockups visar fyllda listor.
Designa en EmptyState-komponent per kontext: "Inga insamlingar än — bli den första", "Du har inte donerat ännu — utforska", etc. Använd varma illustrationer (8-uddiga stjärnor) istället för "no-data"-piktogram.
G2

Laddningstillstånd (loading)

Inga skeletons. Vad ser man medan en sida laddar?
Bygg en <Skeleton>-komponent som matchar varje kort-typ (campaign-card, donor-row, message). Skimmer-animation: linear-gradient(90deg, var(--paper-soft) 0%, var(--paper-deep) 50%, var(--paper-soft) 100%) med translate.
G3

Felfall (error states)

Tre typer av fel saknar design: (1) nätverk dör, (2) server returnerar 500, (3) action timeout.
En <ErrorBoundary> i layout-roten + en <ActionError>-komponent för server actions. Texterna ska vara på svenska, hjälpsamma, inte tekniska. "Något gick fel just nu. Försök igen — eller mejla support@sadaqasweden.se om det fortsätter."
G4

Mobile responsivitet

De flesta sidor har grundläggande media queries men sticky-paneler funkar inte. Donate-card, granskar-rail, wizard-rail — alla blir konstiga på mobil.
På < 768px: (a) Donate-card → fixed bottom sheet, (b) Wizard-rail → horizontellt stepper-band överst, (c) Granskar-kö → swipe-deck mellan ärenden.
G5

Tillgänglighet

Inga ARIA-attribut, ingen explicit fokus-management, kontraster inte mätta.
Audit med axe-core CI på alla routes. focus-visible-stilar på alla interaktiva element. Minimum kontrast 4.5:1 för bodytext (kolla copper-text på cream — det är på gränsen). prefers-reduced-motion respekteras på alla animationer.
G6

Internationalisering

Plattformen är svenskspråkig — bra. Men arabiska ord i community ("Allahumma barik", "Jazak Allah khair") är skrivna inline. Vissa skärmläsare hanterar inte detta väl.
Wrappa arabiska i <span lang="ar">...</span>. Lägg en font-stack för arabiska — t.ex. "Noto Sans Arabic" som fallback.
G7

Tooltip & inline-förklaring

Många koncept förklaras inte: Zakat-OK, Collab, KYC, Trust-poäng, Stripe Connect, Per-fält integritet.
Bygg <HelpTooltip term="zakat-ok" />. Alla termer ligger i lib/content/ordlista.ts. Ikon: liten i i cirkel bredvid termen. Klick: popover med 1-2 meningar + länk till djupare läsning.

§ 03 · VERKLIGHETENSaker som händer i kontakt med riktiga användare

När designen möter användare, lagar och bankerna händer saker som mockups inte fångar.

Lagliga & ekonomiska realiteter

OmrådeRealitetVad detta innebär för designen
Skatteverket Donatorer som vill göra avdrag behöver organisationens 90-konto + verifierat namn. Anonym-flagga får inte gälla för avdragsgilla donationer. Kvittot måste ha personnummer + namn för avdrags-spår.
GDPR "Right to be forgotten" — användare kan kräva radering av allt. Anonymisera donationer (behåll belopp, ta bort kopplingen till profilen). Inte radera helt — det stör revisor-spår. Bygg en GDPR-export-knapp i inställningar.
Stripe Connect Mottagare under 18 år får inte ha eget Stripe-konto. För insamlingar för barn (rullstol-fallet) går pengarna till en förälder eller förening. Tvinga val: "Vem tar emot pengarna?" — privatperson 18+ eller förening.
Penningtvätt (FATF/PMTL) Belopp över viss tröskel triggrar KYC/AML-verifiering. Designen behöver visa en larm-pill: "Belopp över 50 000 kr per donation kräver utökad KYC. Det här hanteras av Stripe automatiskt."
Sanktioner Vissa länder/personer på EU-sanktionslistan får inte ta emot pengar. Innan utbetalning: kör mottagarens namn mot sanktionslistan (OFAC + EU). Om träff → frys + manuell granskning.

Tekniska realiteter

OmrådeRealitetVad designen behöver lägga till
Bilduppladdning Telefoner skickar 12 MB-bilder. R2 kostar att lagra. Klient-side komprimering (max 1920px lång sida, 80% kvalitet). Visa progress-bar. Generera 3 varianter (thumb 200, card 800, full 1600) via Cloudflare Images.
Realtid "Senaste donatorerna" och "Senaste timmen +2 200 kr" antyder live-data. Supabase Realtime subscribe på donation. Throttle till 1 update / 2 sek. Animera in nya rader.
Sökning Fritextsökning över storyn/titeln/orten. Postgres tsvector + tsquery med svenska språk-config. Debounce 300ms. Inga externa sök-tjänster i v1.
Bilduppvisning Inte alla browsers stöder WebP eller AVIF. Använd <Image /> från Next.js (eller Cloudflare Images) som hanterar format-fallback automatiskt.
E-post Donatorer förväntar sig kvitto inom sekunder. Resend webhook efter Stripe-confirmation. Mall i emails/kvitto.tsx (React Email). Retry vid bounce.

Mänskliga realiteter

OmrådeRealitetLösning
Bedrägeri-försök Fejk-insamlingar med stulna bilder kommer testas. Granskning före publicering + omvänd bildsök + dokument-verifiering. Plus: tipsa-knapp på varje insamling.
Granskaren hinner inte 72 h SLA — om kön växer till 50 ärenden hinner inte 3 personer. SLA-overflow-larm i admin. Auto-prioritera små belopp (auto-godkänn med spot-check?) Eller pausa nya insamlingar tills kön är under 20.
Donatorn ångrar sig "Jag donerade fel belopp — ge tillbaka." Stripe-refund tar 5–10 dgr. Klar refund-policy i checkout: "Refund inom 14 dgr — knapp i din mejl-kvitto." Inte chatta med support.
Insamlaren visar inga bevis 3 mån går utan resultat-bevis. Donatorer börjar undra. Auto-påminnelse till insamlaren efter 30/60/90 dgr. Efter 90 dgr utan bevis: flagga publikt ("Bevis försenat") + sänk trust-poäng. Frys nya insamlingar tills bevis kommer.
Insamlingen når inte målet 50 % nått, deadline kommer. Tvinga val vid skapande: "Om vi inte når målet — refund eller kör delvis?" (designen har detta — bra. Men förklara refund-mekaniken tydligare.)

// app/api/stripe/webhook/route.ts
export async function POST(req: Request) {
  const sig = req.headers.get("stripe-signature");
  const event = stripe.webhooks.constructEvent(await req.text(), sig, env.STRIPE_WEBHOOK_SECRET);

  switch (event.type) {
    case "payment_intent.succeeded": {
      const pi = event.data.object;
      // 1. Skapa donation-rad (service_role klient — bypassar RLS)
      await supabaseAdmin.from("donation").insert({
        stripe_payment_intent_id: pi.id,
        belopp_ore: pi.amount,
        insamling_id: pi.metadata.insamling_id,
        donator_id: pi.metadata.donator_id ?? null,
        anonym: pi.metadata.anonym === "true",
      });
      // 2. Uppdatera insamling.insamlat_ore via SECURITY DEFINER-funktion
      await supabaseAdmin.rpc("insamling_inkrementera", { id: pi.metadata.insamling_id, ore: pi.amount });
      // 3. Skicka kvitto-mejl via Resend
      await sendKvitto({ donationId, epost: pi.receipt_email });
      break;
    }
    case "payment_intent.payment_failed": { /* logga + notifiera */ }
    case "charge.refunded": { /* dekrementera insamling, notifiera mottagare */ }
  }
  return new Response("ok");
}

export async function sparaSteg(insamlingId: string | null, steg: number, data: FormData) {
  const { userId } = await kraver(["insamlare"]);
  const schema = stegSchema(steg);            // zod, en per steg
  const parsed = schema.safeParse(objectFromFormData(data));
  if (!parsed.success) return { fel: parsed.error.flatten() };

  const supabase = await createClient();
  if (!insamlingId) {
    // Steg 1 — skapa utkast
    const { data: ny } = await supabase
      .from("insamling")

§ 07 · TREDJEPARTBankID · Stripe Connect · R2 · Resend · Criipto

BankID via Criipto Verify

Vi använder Criipto — inte direkt-integration med BankID. Det är snabbast att komma live.

  1. Användaren klickar "Logga in med BankID" på /login
  2. Vi redirectar till https://login.criipto.com/oauth2/authorize?... med vårt klient-id + redirect-URI
  3. Criipto öppnar BankID-flödet (QR eller på samma enhet)
  4. Användaren signerar i BankID-appen
  5. Criipto callback'ar till /api/criipto/callback?code=...
  6. Vi byter code mot JWT via Criiptos token-endpoint
  7. JWT innehåller sub (en hash av personnumret) + name + given_name + family_name
  8. Vi mappar sub till en profil i Supabase. Om ej finns: skapa
  9. Sätt session via Supabase Auth (custom JWT-token)
lib/criipto/verify.ts
export async function verifieraBankIDToken(token: string) {
  const jwks = createRemoteJWKSet(new URL("https://YOUR-TENANT.criipto.id/.well-known/jwks"));
  const { payload } = await jwtVerify(token, jwks, {
    issuer: "https://YOUR-TENANT.criipto.id",
    audience: env.CRIIPTO_CLIENT_ID,
  });
  // payload.sub är hashen vi använder som persistent ID
  return { sub: payload.sub, namn: payload.name, fornamn: payload.given_name, efternamn: payload.family_name };
}
Integritet

Vi sparar aldrig personnumret i klartext. Bara Criiptos hash (sub-claim). Detta håller GDPR-mässigt. Om Skatteverket eller polisen kräver namn → lagligt grundat utlämnande via Criipto.
